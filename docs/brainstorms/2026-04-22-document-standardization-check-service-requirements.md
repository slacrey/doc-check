# 文档标准化检验服务

## Problem Frame

公司内部多类正式文档存在高频、强规范、强责任的标准化检验需求，包括 AEOS 体系文件、制度文件、新闻宣传稿、党建请示等。当前主要依赖人工逐字核对格式、术语、政治表述、编号、目录页码及缺漏项，单份文件通常耗时 10-20 分钟，且在重复审查中容易出现疲劳漏检。

该需求的目标不是替代最终审核责任，而是建设一个独立的文档标准化检验服务，将确定性强、重复度高、规则可沉淀的检查动作自动化，输出结构化问题清单和修改建议，显著降低人工审核时间并提升一致性。

首期 P0 将以 `AEOS 制度文件` 为试点场景，优先验证规则沉淀、批注回写和审核闭环是否成立，再决定是否向其他文种扩展。

```mermaid
flowchart LR
  A["上传待检文档"] --> B["选择文种 / 规则包"]
  B --> C["文档结构解析"]
  C --> D["格式规范检查"]
  C --> E["内容合规检查"]
  D --> F["问题聚合与分级"]
  E --> F
  F --> G["输出标准化检验报告"]
```

## Requirements

**用户与输入**
- R1. 系统应面向文档起草人员和审核人员提供独立使用入口，支持单份文档上传并发起标准化检验。
- R2. 系统首期 P0 应聚焦 `AEOS 制度文件`，并允许用户在检验前选择对应规则包；后续再逐步扩展到党建请示、新闻宣传稿等其他文种。
- R3. 系统首期仅支持可编辑的 `.docx` 文档检验，暂不纳入其他输入格式。

**格式规范检查**
- R4. 系统应校验模板符合度，包括封面、标题、章节层级、必备章节、页眉页脚、页码、目录与正文页码对应关系等。
- R5. 系统应校验样式规范，包括字体、字号、加粗/斜体、段前段后距、行间距、缩进、对齐方式等版式要求。
- R6. 系统应校验结构和编号连续性，包括章节完整性、章节编号连续性、条款编号连续性、页码连续性及目录映射异常。

**内容合规检查**
- R7. 系统应基于企业术语库对术语使用进行一致性检查，并给出统一用语建议。
- R8. 系统应基于禁用词库、敏感词库和特定标准表述规则，对不合规表述进行拦截和提示。
- R9. 系统应识别常见内容问题，包括中英文标点混用、错别字、基础拼写和基础语法异常。
- R10. 系统应区分确定性错误与建议性提示，避免将低置信度语言问题与硬性规范问题混为一类。

**结果输出与审查支撑**
- R11. 系统应输出结构化《标准化检验报告》，按问题类型、严重级别和文档位置进行汇总。
- R12. 每条问题应包含明确定位信息和可操作建议，定位至少精确到页码、章节或段落，并尽可能附带原文片段。
- R13. 报告应标注所依据的规则来源或规则名称，便于审核人员追溯判断依据。
- R14. 系统应保留本次检验所使用的规则包版本、检验时间和文档标识，支持后续审计和结果复核。
- R14a. 系统首选应输出回写批注版 `.docx` 文档，将问题以批注形式直接标记在原文对应位置，并保留汇总视图能力。

**规则治理**
- R15. 系统应支持规则来源的结构化维护，至少覆盖规范文档、企业术语库、禁用词库三类规则资产。
- R16. 系统应支持按文种维护不同规则包，并允许同一基础规则被多个文种复用。
- R17. 系统应支持对个别问题进行“误报”或“可接受例外”标记，作为后续规则优化的输入，但首期不要求形成自动学习闭环。

**安全与审计**
- R18. 系统应满足企业文档保密要求，对上传文档、检验结果和规则资产提供访问控制、留痕和可配置的数据保留策略，并支持私有化部署。

## Success Criteria

- 将单份标准化检验平均耗时从 10-20 分钟压缩到 2-5 分钟以内，人工主要投入转为复核与最终判断。
- 在首批 `AEOS 制度文件` 试点中，系统能够稳定发现大部分确定性格式问题和高频内容规范问题，避免明显漏检。
- 检验报告对审核人员是可执行的，能直接支撑修改或退回，不需要再额外人工整理问题清单。
- 不同审核人员针对同一份文档的标准化结论差异显著收敛。
- 试点用户能够直接基于回写批注版 `.docx` 完成复核和修改，不需要再手工将报告内容二次转录回原文。

## Scope Boundaries

- 首期以独立工具模块提供服务，不与现有流程审批系统集成。
- 首期输出以回写批注版 `.docx` 为主，不要求系统直接自动改正文档正文。
- 首期不以扫描件、图片型 PDF、手写批注等 OCR 场景为范围重点。
- 首期不做跨文档比对、相似制度查重或知识问答类能力。
- 首期不替代最终人工审核责任，尤其是涉及政策口径、政治判断或复杂上下文的内容合规结论。

## Key Decisions

- **先做规则驱动型检验，再逐步引入更复杂能力**: 当前需求中的大部分高价值场景都属于强规则、可解释、可审计的问题，适合优先通过规则包沉淀。
- **按文种建设规则包，而不是做一个“大而全”的统一规则集**: 党建请示、制度文件、新闻稿的标准差异较大，按文种拆分规则更容易控制准确率和维护成本。
- **首期 P0 只打 `AEOS 制度文件`**: 该场景规则更稳定、样本更集中、格式检查价值更明确，适合作为第一批落地对象验证准确率和流程闭环。
- **将格式检查和内容检查分层输出**: 格式问题更适合确定性判定，内容问题存在置信度差异，分层有助于用户理解和处置。
- **将报告可追溯性作为核心能力，而不是附属能力**: 对企业正式文件的审查，能解释“为什么判错”与“依据什么规则”同样重要。
- **首期输入格式收敛为 `.docx`，输出以批注版文档为主**: 这能显著降低解析复杂度，并把结果直接落在用户最熟悉的修订载体上。
- **部署方式以可私有化为前提，而不是强制内网限定**: 先保证部署灵活性，再按企业实际安全要求落地环境边界。

## Dependencies / Assumptions

- 已存在相对权威且可供整理的《AEOS 文件标检标准作业》、术语库、禁用词库及文种模板样例。
- 各业务归口部门愿意持续参与规则维护，至少提供规则确认、样例文件和误报反馈。
- 首期优先支持可编辑文档格式，文档内容能被稳定提取到结构化层级。
